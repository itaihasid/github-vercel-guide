Part 01

What are GitHub & Vercel, and why do you need them?

Before doing anything, let's understand what these tools actually are — in plain English.

The Kitchen Analogy

GitHub is your recipe notebook — it stores every version of your project, so you can always go back to an earlier draft. It lives in the cloud, so nothing is ever lost.

Vercel is the restaurant that serves your food to the world. You hand it your recipe (from GitHub), and it automatically opens to customers 24/7. Every time you update your recipe, the restaurant immediately serves the new version.

🗂 GitHub

Stores all your files and their full history. If you break something, you can rewind. Think of it as Google Drive, but much smarter.

⚡ Vercel

Takes your files from GitHub and turns them into a live website — with a real URL anyone can visit, completely free.

🔄 The Magic Link

Connect them once. After that, every time you save a change to GitHub, your live website updates automatically.

💰 The Cost

Both are free for personal projects. You can host a full website with a real URL, paying nothing.

Bottom line: GitHub is your safety net and storage. Vercel is your publisher. Together they replace needing a server, manual file uploads, or any DevOps knowledge.
Part 02

Setting Up Your Accounts

You only do this once. It takes about 5 minutes total.

1

Create a GitHub account

Go to github.com and sign up with your email. Choose a username — it will appear in your project URLs, so pick something sensible.

Your username becomes part of your address: github.com/your-username
2

Create your first Repository

A repository (repo) is just a project folder on GitHub. One project = one repo.

On GitHub
Click + New Repository
Name it: my-website
Select: Public (so Vercel can read it)
Tick: Add a README file
Click: Create Repository
3

Upload your project files

No software needed. GitHub lets you drag-and-drop files right in the browser.

In your new repo
Click Add file → Upload files
Drag your HTML, CSS, images into the box
Type a message: "First upload"
Click: Commit changes
What's a "Commit"? It's saving with a label. Every save is tracked — so you can always see what changed, and when, and revert if needed.
4

Create a Vercel account

Go to vercel.com and click "Continue with GitHub". That's it — Vercel and GitHub are now connected. One click, no separate password needed.

Part 03

Deploying Your Website

This is the step where your files become a real, live website with a URL you can share.

1

Import your project into Vercel

On vercel.com
Click Add New Project
Click Import Git Repository
Your GitHub repo appears in the list → click Import
2

Configure (or just skip it)

For a basic HTML/CSS site, change nothing. Just scroll down and click Deploy. Vercel auto-detects what kind of project you have.

Vercel is smart. It recognises plain HTML sites, React apps, Next.js — and adjusts its settings automatically.
3

Watch it go live 🎉

Within 10–30 seconds, your site is live. You'll get a free URL:

Your live URL
https://my-website-abc123.vercel.app

You can also connect a custom domain like myname.com for free in Vercel settings.

Part 04

Updating Your Website

Once it's set up, updating is the easiest part. Change a file, and your site updates. That's it.

The Magic Part

GitHub and Vercel watch each other. The moment you save (commit) a change to GitHub, Vercel automatically rebuilds and publishes your site. No clicking "upload". No FTP. No server access. It just happens.

1

Edit a file directly on GitHub

Browser method — no software needed
Open your repo on github.com
Click the file you want to edit (e.g. index.html)
Click the ✏️ pencil icon (top right)
Make your changes
Add a short note: "Fixed the headline text"
Click Commit changes
2

Vercel rebuilds automatically

Go to your Vercel dashboard. You'll see a new deployment starting within seconds. Within a minute, your live site reflects the change.

Preview URLs: Every commit also creates a temporary preview link so you can check changes before they go fully live — great for catching mistakes first.
3

Made a mistake? Roll back instantly.

In your Vercel dashboard, every previous deployment is listed. Click any older version and hit "Promote to Production" — your site reverts immediately. No panic required.

Part 05

Using Other People's Code from GitHub

One of GitHub's superpowers: millions of free, open-source projects you can copy, customise, and deploy as your own.

What is "Forking"?

When you Fork someone's GitHub project, you make your own personal copy of it — in your own account. You can edit it however you want without affecting the original. It's like photocopying a recipe and putting your own twist on it.

1

Find a project you like

Browse github.com/explore or search for what you need. Look for repos with many ⭐ stars — that indicates popularity and quality.

Example: want a portfolio template? Search "portfolio template HTML" on GitHub and you'll find hundreds of free options ready to customise.
2

Fork it to your account

someone-else / cool-portfolio-template ⑂ Fork
⭐ 2,400 stars🍴 843 forks👀 Public
index.htmlstyle.cssREADME.md

Click the Fork button at the top right of any repo. GitHub creates a copy under your account: your-username/cool-portfolio-template

3

Customise it

Edit any files in your forked copy — change text, colours, images, whatever. The original stays untouched. Only your copy changes.

Always check the licence first. Look for a file called LICENSE in the repo. MIT or Apache licences mean use freely. Keep any required attribution credit in place.
4

Deploy it on Vercel

Since the fork is now in your GitHub account, go to Vercel → Add New Project → import your forked repo. It deploys exactly like your own project. Live in under a minute.

🔍 Where to find good repos

github.com/explore, github.com/topics, or Google "awesome [topic] github" for curated lists.

⭐ What to look for

Many stars, recent commits (updated recently), and a README with clear instructions.

Part 06

Working with Claude + Folders

How to use Claude to build things, keep your project organised, and get your code into GitHub and live — without writing a single line yourself.

The New Way to Build

You describe what you want to Claude. Claude writes the code. You paste it into the right folder. GitHub stores it. Vercel publishes it. You made every decision — Claude just handled the technical part.

1

Set up your folder structure before asking Claude anything

A clean folder layout matters. Vercel expects your homepage at the top level, and Claude writes much better code when it knows exactly where each file lives.

Standard folder structure
my-website/ ← root (top level of your repo)
  index.html   ← homepage — must be here
  about.html   ← other pages
  css/
    style.css    ← all styling in here
  js/
    script.js    ← any interactions
  images/
    logo.png     ← your images
  README.md    ← describe your project
Critical: index.html must sit at the root (top level) of your repo — not inside any subfolder. Vercel uses it as the entry point. If it's buried, your site won't load.
2

Ask Claude the right way

Vague requests produce generic code. Specific requests produce code that plugs straight into your structure.

❌ Too vague
"Make me a contact form"
✅ Specific
"Write a contact form for index.html. Styles in css/style.css. Fields: name, email, message. Dark background, white text, minimal."
Always tell Claude: the filename, where it lives, which other files it connects to, and the visual style. Claude will write code that fits your exact setup.
3

Use Claude Projects to keep context across sessions

Claude's Projects feature is a persistent workspace — Claude remembers your setup between conversations. Essential when you're building something over multiple sessions.

Setting up a Claude Project
1. In Claude → click Projects in the left sidebar
2. Create a new project, name it after your site
3. Open Project Instructions and paste something like:

"I'm building a website called [name]. Main file is index.html at the root. CSS in css/style.css. JS in js/script.js. It's a [e.g. portfolio for a photographer, dark theme, minimal style]. Always write code that fits this structure."

4. Every conversation in this Project now starts with Claude already knowing your setup.
4

Copy Claude's code into the right file on GitHub

Claude tells you which file the code belongs to. Navigate to that file in your repo and paste it in — that's the whole workflow.

Step by step
Claude writes code → says it goes in css/style.css
→ Open your GitHub repo
→ Navigate to css/ → click style.css
→ Click the ✏️ pencil icon to edit
→ Select all → paste Claude's code
→ Write a commit message: "Updated styles from Claude"
→ Click Commit changes
→ Vercel rebuilds → live in ~30 seconds
One file at a time. If Claude gives you HTML and CSS in the same reply, paste each piece into its own file separately. Mixing them together causes problems later.
5

When something breaks — give Claude the full picture

Claude can fix its own mistakes, but only if you paste in the relevant code and describe exactly what's happening versus what you expected.

❌ Not helpful
"The site is broken, fix it"
✅ Helpful
"The navbar disappears on mobile. Here's my css/style.css: [paste]. Here's index.html: [paste]. Desktop looks fine. On my phone the menu isn't visible."
Golden rule: Claude can't see your files — you have to bring them into the conversation. The more context you show, the faster it fixes.

📁 One task = one file

Ask Claude for one file at a time. Focused requests produce cleaner code than asking for a whole site in one go.

🏷 No spaces in filenames

Use lowercase with hyphens: about.html not About Page.html. Spaces cause subtle bugs that are annoying to track down.

💬 Start with context

At the start of any task, briefly say: "I'm building a portfolio, dark theme, file structure we set up." Even in a Project, a quick recap helps Claude stay accurate.

🔁 Iterate, don't restart

If Claude's output is 80% right, paste it back and say "fix just the mobile menu." Small corrections beat starting over every time.
